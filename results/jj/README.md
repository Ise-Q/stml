# Results Artifacts

This directory contains machine-readable artifacts generated by the signal
characterization and replication pipeline. The reports in `reports/` render
these numbers for human reading; the files here are the structured records.

## Files

`thresholds.json`
: Train-only baseline metrics and suggested G1 cutoffs. These are produced by
`stml.replication.run_characterize`.

`ledger.json`
: Trial-level audit log for the replication search. Each evaluated parameter
configuration is recorded with its family, cell, tier, parameters, train
discrepancy, validation metrics, gate result, and provenance note.

`top_candidates.json`
: Compact list of the best passing candidate per passing family. This is the
most convenient structured summary if you want to inspect the final selected
replicas programmatically.

## thresholds.json

The thresholds file is deliberately train-only. Its provenance block records:

- `calibration_window`: always `train` for the canonical run;
- `train_date_range`: `2020-01-03` to `2021-07-01`;
- `floor`: the post-embargo validation `n_eff` floor, currently `10`;
- `convention`: `next_day`;
- `cutoff_margin`: `0.05`;
- `cutoff_rule`: strongest no-skill baseline plus margin.

For each instrument and asset class, the file contains:

`baseline_metrics`
: Scores for naive predictors on the train split.

`suggested_cutoffs`
: The G1 pass-bar before the search-width margin is added.

`persistence_reference`
: Score of the one-step persistence baseline, `s_t = s_{t-1}`. This is reported
as a reference, not used as the pass bar.

`n_train`
: Number of train rows used for that instrument or pooled class.

The no-skill baselines used for the cutoff are:

- `always_flat`: predict `0` everywhere.
- `majority_class`: always predict the most common train label.
- `stratified_random`: sample labels from the train label distribution with
  fixed seed 0.

For each metric:

```text
suggested_cutoff = max(always_flat, majority_class, stratified_random) + 0.05
```

G1 then applies an additional multiplicity margin:

```text
margin = 0.02 * (1 + log1p(n_configs) / 10)
```

So the actual G1 comparison is:

```text
validation_metric > suggested_cutoff + margin
```

This keeps validation passes from being explained by class imbalance or by
searching many configurations.

## ledger.json

The ledger is the audit trail. It answers:

- which family was searched;
- which cell was searched;
- whether the tier was `tpe` or `grid`;
- which parameters were tried;
- what train discrepancy was achieved;
- what validation metrics were observed;
- whether the selected winner passed G1-G4;
- what prior trials motivated the run.

Important fields:

`archetype`
: The family name, such as `mean_reversion` or `vol_regime_gated`.

`cell`
: Instrument or pooled cell, such as `es1s` or `pool:energy`.

`tier`
: `tpe` for Optuna-guided search above the `n_eff` floor, `grid` for exhaustive
deterministic search below it.

`params`
: Parameter values evaluated in that trial.

`train_discrepancy`
: Search objective value. Lower is better. It equals
`1 - composite_skill`, where `composite_skill = 0.5 * (kappa + ordinal_skill)`.

`val_metrics`
: Validation metrics recorded for reporting and gates. These do not choose the
winner.

`gate_result`
: G1-G4 booleans and supporting diagnostics for the final selected trial.

`motivated_by`
: Provenance note that links a trial to prior best trials or records that this
was the first iteration.

## top_candidates.json

This file contains one best passing cell per passing family. A family can pass
multiple cells, but this artifact keeps the highest validation composite among
its passing cells.

Fields:

`archetype`
: Passing family.

`cell`
: Best passing cell for that family.

`params`
: Selected parameter set.

`tier`
: Search tier used for that cell.

`n_eff`
: Post-embargo validation effective sample size for the cell.

`val_metrics`
: Validation metrics for the selected candidate.

`gates`
: Final G1-G4 pass/fail state.

## How These Files Relate To The Reports

- `reports/signal-characterization.md` explains the threshold calibration and
  `n_eff` pooling map rendered from `thresholds.json` and characterization code.
- `reports/<family>.md` uses `ledger.json` style trial/gate information to show
  per-family outcomes.
- `reports/replication-summary.md` summarizes family passes and selected
  candidates, consistent with `top_candidates.json`.

## Reading Pass/Fail Values

A `passed: true` candidate means all four validation gates cleared:

1. G1: validation kappa and ordinal skill beat train-only chance cutoffs plus
   search-width margin.
2. G2: validation skill is positive and retains at least half the train skill.
3. G3: nearby numeric parameter settings still form a validation plateau.
4. G4: kappa, ordinal skill, and NAV increment correlation all agree in sign.

The held-out test block is not used to produce these JSON search artifacts. It
is accessed only once in the final confirmation stage rendered in
`reports/replication-summary.md`.
