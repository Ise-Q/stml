# Reading The Signal Replication Reports

This directory contains the human-readable artifacts for the primary-signal
characterization and replication work. The reports assume a few project-specific
terms, so this README explains the workflow, terminology, and reasoning behind
the validation logic.

It also contains [`feature-catalog.md`](feature-catalog.md), the documentation
for the metamodel **feature-engineering layer** (the next phase, built on top of
the replication study's findings). See the final section, "Feature Catalog".

## Recommended Reading Order

1. `signal-characterization.md`
   - Describes the released signal itself: lead/lag behavior, alpha style,
     base-rate drift, cross-asset structure, effective sample size, and
     train-only threshold calibration.
2. `<family>.md`, for example `mean_reversion.md`
   - Shows one replication family at a time: parameter space, best parameters by
     cell, validation metrics, and G1-G4 gate diagnostics.
3. `replication-summary.md`
   - Summarizes which families/cells passed, search coverage, pooling behavior,
     caveats, and the one final held-out test confirmation.
4. `replication-ledger.md`
   - Compact table of evaluated trials recorded during the search.
5. `missing-data-report.md`
   - Data-cleaning context for the futures OHLCV panel.
6. `feature-catalog.md`
   - The metamodel feature-engineering layer: every engineered, regime, latent,
     and cross-sectional feature column, with its lookback, leakage class, and
     reuse pointer. See the "Feature Catalog" section below.

## Workflow

The replication workflow is:

1. Load cleaned OHLCV and released primary signals.
2. Split the released signal panel chronologically.
3. Characterize the signal and compute train-only baseline thresholds.
4. Define candidate rule families, called archetypes.
5. For each `(family, cell)`, search parameters on train only.
6. Evaluate the selected winner on validation through G1-G4 gates.
7. After gates are frozen, touch the held-out test block once for final
   confirmation.

The released signal panel runs from `2020-01-03` to `2022-06-30` and is split
60/20/20:

| split | date range | purpose |
|---|---|---|
| train | `2020-01-03` to `2021-07-01` | parameter search and threshold calibration |
| validation | `2021-07-02` to `2021-12-30` | G1-G4 gate checks |
| test | `2021-12-31` to `2022-06-30` | one final confirmation after selection |

Older OHLCV history is not used as extra labelled signal training data. The
replication runner keeps OHLCV from `2019-01-01` onward so rolling features have
warm-up history before the 2020 signal era. The released signal labels themselves
start in 2020.

## Core Terms

`target`
: The released primary signal for one instrument, with values in `{-1, 0, +1}`.

`replica`
: A candidate signal generated by a rule family from OHLCV. The goal is to
match the target signal's timing and direction.

`family` or `archetype`
: A rule template such as `mean_reversion`, `ts_momentum`,
`breakout_donchian`, `vol_regime_gated`, `hybrid_filtered_momentum`, or
`xsect_rank`.

`cell`
: The unit being replicated. Usually this is one instrument, such as `es1s`.
The special cell `pool:energy` combines thin energy instruments.

`parameter axis`
: One tunable dimension inside a family, such as `lookback`, `z_window`, or
`deadband`.

`full grid`
: The Cartesian product of all parameter axes for a family. For
`mean_reversion`, the grid is `lookback x z_window x deadband`.

`Optuna`
: The optimization library used to search parameters.

`TPE`
: Tree-structured Parzen Estimator, Optuna's guided sampler used here. It
chooses parameter combinations, observes train discrepancy, and biases later
trials toward more promising regions.

`budget`
: Number of Optuna/TPE trials per `(family, cell)`. The canonical run uses
`budget = 64`.

`coverage`
: `min(budget, full_grid_size) / full_grid_size`. This is an approximate
budget-to-grid ratio, not a guarantee that TPE visited every unique grid point.

`embargo`
: Rows removed at both edges of the validation window so a long unchanged signal
run cannot leak across train/validation or validation/test boundaries.

`post-embargo`
: After the validation-edge embargo has been applied.

`n_eff`
: Effective sample size. It counts constant signal runs, not daily rows. A
40-day constant `+1` stretch is one independent regime call, not 40 independent
observations.

`n_eff floor`
: Minimum post-embargo validation effective sample size for a standalone
verdict. The current floor is `10`.

`gated`
: Evaluated through the G1-G4 robustness gates.

`base-rate drift`
: A change in class frequencies across splits. For example, a signal may be
mostly flat in train and much more active in validation. G2 handles this by
comparing skill against each split's own baseline.

## Why n_eff Matters

The signal is piecewise constant. If an instrument holds the same value for many
days, those daily rows are highly dependent. Counting rows would exaggerate how
much independent evidence we have.

Example:

```text
[1, 1, 1, 0, 0, -1, -1, -1, 1]
```

This has four constant runs:

```text
[1,1,1] -> [0,0] -> [-1,-1,-1] -> [1]
```

So `n_eff = 4`, even though there are 9 rows.

In the pipeline, `n_eff` is used to:

- decide whether an instrument can be judged standalone;
- route low-power instruments into an asset-class pool;
- choose TPE search above the floor and deterministic exhaustive grid below it;
- communicate how much independent validation evidence supports a pass/fail.

## Search Logic

For each `(family, cell)`, the search objective is train-only discrepancy:

```text
composite_skill = 0.5 * (kappa + ordinal_skill_vs_flat)
discrepancy = 1 - composite_skill
```

Lower discrepancy is better. Validation metrics are recorded, but they do not
choose the winner.

When `n_eff >= 10`, the search uses seeded Optuna TPE with `budget = 64`.
When `n_eff < 10`, the search uses deterministic exhaustive grid evaluation.

Example: `mean_reversion` has:

```text
lookback:  [5, 10, 20, 40]
z_window:  [20, 40, 60]
deadband:  [0.0, 0.25, 0.5, 1.0, 1.5]
```

Full grid size is `4 * 3 * 5 = 60`. A budget of 64 is enough to cover a grid of
this size in budget terms, but TPE still samples trial-by-trial and may revisit
equivalent configurations.

## G1-G4 Gates

A candidate passes only if all four gates pass.

### G1: Beats Baseline Plus Multiplicity

G1 asks whether validation agreement is better than simple no-skill behavior.
Thresholds are calibrated on train only from naive baselines:

- `always_flat`: predict `0` every day.
- `majority_class`: always predict the most common train label.
- `stratified_random`: sample labels from train label frequencies with seed 0.

For each metric:

```text
cutoff = max(no-skill baseline scores) + 0.05
```

G1 then adds a search-width margin:

```text
margin = 0.02 * (1 + log1p(n_configs) / 10)
```

With `n_configs = 64`, the margin is about `0.028`.

G1 requires both validation `kappa` and validation `ordinal_skill` to exceed
their cutoffs plus this margin.


The constants in this gate are checkpoint heuristics, not statistically
identified truths. The `0.05` cutoff margin is an effect-size buffer: it says a
replica should clear the best no-skill baseline by a visible amount, not by a
rounding error. It is reasonable because `kappa` and `ordinal_skill` are
chance-corrected, where `0` means roughly no skill, but it is still manual.

Risks of the fixed `0.05` margin:

- it may be too lenient when post-embargo `n_eff` is very small;
- it may be too strict when `n_eff` is large;
- it does not explicitly model metric uncertainty.

Better alternatives would be a sensitivity sweep over margins such as `0.025`,
`0.05`, `0.075`, and `0.10`, or an empirical null where signal runs are
block-shuffled and the full search is rerun to estimate how often a fake signal
can pass.

The search-width margin is also heuristic:

```text
margin = 0.02 * (1 + log1p(n_configs) / 10)
```

`log1p(n_configs)` means `log(1 + n_configs)`. The `0.02` is the base extra
margin. Dividing by `10` dampens the log term so the penalty grows slowly rather
than dominating the gate. This is a mild multiplicity penalty, not a formal
Bonferroni correction.

Typical values:

| n_configs | margin |
|---:|---:|
| 1 | 0.021 |
| 8 | 0.024 |
| 64 | 0.028 |
| 512 | 0.032 |
| 1000 | 0.034 |

This margin is not the Optuna objective. Optuna searches by train discrepancy.
The margin is applied later, during G1 validation.

### G2: Drift-Aware Generalization

G2 asks whether train skill transfers to validation after adjusting for each
split's own majority baseline.

Pass rule:

```text
skill_val > 0
skill_val >= 0.5 * skill_train
```

This blocks a replica that only follows a drifting majority class.


The `0.5` retention fraction is also a heuristic. It means validation skill must
retain at least half of train skill. Requiring `skill_val >= 1.0 * skill_train`
would usually be too strict because the train winner is selected by search and
is therefore upward-biased on train. The current rule accepts degradation, but
not complete collapse.

Risks:

- it can be too lenient when train skill is tiny;
- it can be too strict when the signal regime genuinely shifts;
- ratio rules are unstable when `skill_train` is near zero.

Better alternatives would use rolling train folds, uncertainty intervals, or a
lower-confidence-bound criterion such as `mean_skill - c * standard_error`.

### G3: Parameter Plateau

G3 asks whether the winner is robust to small parameter nudges. The code builds
`+/-1 step` neighbors along numeric/ordinal axes and recomputes validation
composite skill for each neighbor.

Pass rule:

```text
min(neighbor_composites) > cutoff
std(neighbor_composites) < 0.15
```

Categorical strategy switches such as `base`, `regime`, and `score` are held
fixed because flipping them changes the strategy type rather than nudging a
numeric parameter.


The `std < 0.15` tolerance asks whether nearby parameter settings produce
roughly similar validation composite scores. It is not enough by itself: the
worst neighbor must also clear the cutoff. A low standard deviation around bad
scores is still a fail.

Example of the desired behavior: `mean_reversion / si1s`.

- winner validation composite: `0.362`;
- neighbor composite range: `0.224` to `0.418`;
- neighbor standard deviation: `0.063`;
- final test composite: `0.169`.

This is a plausible plateau. Nearby settings remain positive and reasonably
close, and the held-out confirmation remains positive, though weaker.

Example of the rejected behavior: `vol_regime_gated / gc1s`.

- winner validation composite: `0.479`;
- neighbor composite range: `-0.016` to `0.529`;
- neighbor standard deviation: `0.166`;
- G1, G2, and G4 passed, but G3 failed.

This is the kind of knife-edge G3 is meant to catch: the selected parameter set
looks strong, but a small parameter change can produce a negative score and the
neighborhood spread is wide.

The exact `0.15` cutoff is a calibration choice. The summary report checks
`0.10`, `0.15`, and `0.20`; the number of passing families stays at `3`, which
supports the stability of the headline conclusion. It does not prove that
`0.15` is uniquely optimal.

### G4: Multi-Metric Consistency

G4 checks that the validation result is not carried by one metric while other
views disagree.

Pass rule:

```text
kappa > 0
ordinal_skill > 0
NAV increment correlation > 0
```

## Metric Glossary

`kappa`
: Cohen's kappa. Agreement beyond chance. It is useful when classes are
imbalanced because a high raw accuracy can be meaningless if both series are
mostly flat.

`ordinal_skill`
: Chance-corrected directional skill. The labels are ordered as `-1`, `0`, `+1`.
A full sign flip, `-1` versus `+1`, is penalized more than missing participation
by predicting `0`.

`NAV increment correlation`
: Correlation between per-day PnL increments of replica and target:
`replica_signal_t * next_day_return_t` versus
`target_signal_t * next_day_return_t`.

`MCC`
: Matthews correlation coefficient. A correlation-like classification metric
that is robust to imbalance. It is included in threshold calibration and
diagnostics.

`macro_f1`
: Average F1 score over classes present in the true signal. It keeps minority
classes visible.

`balanced_acc`
: Mean recall across classes. It avoids letting the majority class dominate the
score.

`composite_skill`
: The search objective's skill measure: `0.5 * (kappa + ordinal_skill)`.


The `0.5` is just an average:

```text
composite_skill = (kappa + ordinal_skill) / 2
```

It keeps the score on the familiar scale where `1` is perfect, `0` is
chance-like, and negative values are worse than chance. Multiplying by `0.5`
does not materially change the ordering of candidates compared with using
`kappa + ordinal_skill`; it makes the objective easier to interpret and keeps
`discrepancy = 1 - composite_skill` clean.

The risk is that an average can hide metric disagreement. That is why G1 and G4
also check the individual metrics instead of trusting the composite alone.


## Calibration Status And Recommended Improvements

The current gate constants are transparent and conservative enough for a
checkpoint replication study, but they are not fully validated statistical
cutoffs. The most important limitations are:

- `0.05` in G1 is a manual effect-size buffer;
- the G1 search-width margin is only a mild heuristic multiplicity penalty;
- G2's `0.5` retention rule is an engineering compromise;
- G3's `0.15` standard-deviation tolerance is a robustness heuristic;
- the composite objective averages two metrics with equal weight by design.

Recommended follow-up validation:

1. Sweep `cutoff_margin`, G1 margin strength, G2 retention fraction, and G3
   standard-deviation tolerance.
2. Use block/run-level permutation tests to estimate how often the full search
   can manufacture a fake pass.
3. Consider nested train validation for parameter selection before touching the
   report validation split.
4. Consider a robustness-aware objective, for example mean train-fold skill
   minus instability across folds.
5. Report confidence intervals or bootstrap ranges for validation metrics when
   post-embargo `n_eff` is small.

## Pooling Logic

The energy members `cl1s`, `ho1s`, and `ng1s` have post-embargo validation
`n_eff` values of `9`, `9`, and `2`, all below the standalone floor of `10`.
They are therefore pooled as `pool:energy`.

The pool's effective sample size is the sum, `9 + 9 + 2 = 20`, because the
cross-asset signal correlations are low enough that the regime calls are treated
as near-independent. The pool is evaluated with per-member metrics first and
then equal-weight averages. This avoids a false pass caused by concatenating
different instruments and accidentally matching their base rates.

## How To Interpret A Pass

A pass means:

- parameters were selected on train only;
- validation cleared G1-G4;
- the result showed agreement beyond simple baselines;
- nearby numeric parameter settings did not immediately collapse;
- directional and PnL-shaped metrics agreed in sign.

A pass does not prove the original production model was recovered exactly. It is
evidence that the family can reproduce the released signal's behavior on that
instrument or cell under the stated validation protocol.

## Feature Catalog (Metamodel Feature-Engineering Layer)

`feature-catalog.md` documents the feature-engineering layer built on top of the
replication findings — the input features for the triple-barrier metamodel. It is
generated by `stml.metamodel.build_features` (via `stml.metamodel.catalog`) and is
the graded methodology artifact for that phase. The structured matrix and maps it
describes live in [`../results/`](../results/README.md); the code lives in
[`../src/README.md`](../src/README.md).

The catalog has one row per produced feature column, grouped by family, with:

`family`
: F1 counter-trend, F2 vol/dispersion, F5 signal-derived, F6 momentum-contrast,
F7 microstructure, F10 OHLC price-action, F8 calendar, F3 regime, F4 latent,
F9 cross-section, F11 cross-asset macro context.

`what_it_captures`
: A one-line description of the feature's economic/statistical content.

`lookback`
: The trailing window (e.g. `20d`, `expanding`, `n/a`).

`leakage_class`
: **E** engineered (causality proven by truncation-invariance), **TF** fitted
(causality proven by a fit-on-FE-train provenance assertion), **LI** the
label-interface subset (σ and trailing run-length) earmarked for the deferred
triple-barrier label.

`reuse_pointer`
: Where the feature's logic comes from (e.g. `na_checks.rolling_vol`,
`archetypes._score_mean_reversion`, `fit: FE-train class-pooled`).

The catalog also records, as prose notes, the load-bearing methodology points: F1
is the highest-value family per the C1 characterization (the signal is
short-horizon mean-reversion); F9 is expected-negative (cross-asset mean
`|corr| ~ 0.09`); the F5 trailing run-length is causal on `s[:t+1]` and distinct
from the full-period `run_length_p90` used only for embargo sizing; the F7 Amihud
illiquidity uses a zero-volume → NaN guard; the F11 cross-asset macro family is
point-in-time publication-lagged per release class (daily market series at their
own close = lag 0; weekly EIA inventories at the Friday stamp + 6 calendar days;
monthly ISM/Caixin PMIs at the month-end stamp + 1 business day), z-scored on the
FE-train slice of the macro panel reindexed onto the equity trade-date cadence and
broadcast identically to all 11 instruments; and the autoencoder-vs-PCA(k=4)
reconstruction MSE per asset class (the autoencoder is retained regardless — there
is no performance gate, grading is methodology).

The F11 macro family is also persisted standalone to
`data/macro_features_engineered.{parquet,csv}` (keyed `date`,`instrument`,
row-aligned to the matrix).

### Causality contract

Every feature is computed at a non-zero-signal trade day using only information
`<= t`. Fitted models (GMM, Markov, PCA, KMeans, autoencoder, scalers, and the F11
macro z-scorer) are trained on the feature-engineering train partition only —
`FE-train`, ending `2021-07-01` —
and the matrix carries a `partition` column and a constant `fe_train_end_date`
column so the downstream model phase can honor that boundary. Two separate proofs
back this up: truncation-invariance for engineered (E) features and a fit-provenance
assertion (training indices and standardization stats ⊆ FE-train) for fitted (TF)
features, because a frozen transform is truncation-invariant even if mis-fit, so
truncation alone cannot prove fit-on-train.
